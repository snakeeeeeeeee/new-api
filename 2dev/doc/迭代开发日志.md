## 2026-04-30 | 聚合分组请求标识亲和策略

- [x] 聚合 route affinity 策略化
  - 实现内容：
    - `aggregate_groups` 新增 `route_affinity_strategy`，默认 `platform_user`，兼容旧聚合分组。
    - `aggregate_groups` 新增 `route_affinity_key_sources` 文本 JSON 字段，用于配置请求标识提取来源。
    - 支持 `platform_user`、`request_first`、`request_only`、`off` 四种策略。
    - `request_first` 优先提取请求标识，提取不到时回退平台账号 ID。
    - `request_only` 只使用请求标识；请求里没有标识时不读写 route affinity，当前请求按权重选路。
    - 请求标识来源复用 channel affinity 的提取能力，并补充 `header` / `query` 来源类型。
    - 默认来源包含 `X-Aggregate-Affinity-Key`、`aggregate_route_affinity_key` query、`metadata.aggregate_route_affinity_key`、`metadata.user_id`、`prompt_cache_key`、`user`、`cachedContent`。
    - route affinity cache 仍使用 `new-api:aggregate_route_affinity:v3`，默认 `platform_user` key 形态保持旧逻辑；请求标识 key 使用来源描述和 SHA1 短指纹，避免把原始用户标识直接写进 cache key。
    - `admin_info.aggregate_route_affinity` 增加策略、来源类型、来源 key/path、key hint、key fp，便于从日志判断本次是否真的使用了请求标识。
    - 聚合分组编辑面板 Cluster tab 增加亲和策略下拉；请求标识来源编辑收进“高级：自定义请求标识来源”，默认不展开，避免普通操作员直接面对 Header / Query / JSON path 配置。

## 2026-04-30 | 聚合分组客户端专用流量池 v1

- [x] Claude Code CLI 定向池
  - 实现内容：
    - `aggregate_groups` 新增 `client_route_pools` 文本 JSON 字段，空配置等价关闭。
    - 创建、编辑、详情、列表接口读写 `client_route_pools`。
    - 配置校验覆盖负权重、重复 target、真实分组不存在、引用聚合分组等场景。
    - Claude Code CLI 识别条件为 `/v1/messages`、`claude-*` 模型、`User-Agent` 包含 `claude-cli/`；`X-App=cli`、`Anthropic-Beta` 包含 `claude-code`、`metadata.user_id` 作为辅助观测特征。
    - `failover` / `cluster` 两种模式下，CLI 请求都会优先进入 `claude_code_cli` 专用池。
    - Failover 下专用池按自己的 target 顺序形成独立链路；Cluster 下专用池按自己的 target 权重分发。
    - 专用池候选复用当前模式的模型支持、手动禁用过滤和智能策略。
    - 专用池内当前子分组先按 `RetryTimes` 组内重试，耗尽后切同池其他 target。
    - 专用池不可用或耗尽后按 `fallback_to_default` 回退当前模式的默认流量池或失败。
    - Cluster 下专用池 route affinity 使用 `aggregate_group + pool + user_id` 独立 key，不污染普通 cluster 亲和。
    - Failover 下专用池使用独立运行态 key，不污染普通 failover 链路起点。
    - CLI fallback 到默认池时不写普通 cluster affinity，也不更新默认 failover 运行态，避免临时兜底改变普通请求落点。
    - 消费日志 / 错误日志 `admin_info` 增加 `client_type`、`client_route_pool`、`client_route_target`、`client_route_fallback`、`aggregate_route_pool`。
    - 运行态接口新增 `runtime.client_route_pools[]`，单独返回专用池节点、权重、有效权重、RPM 和降级状态。
    - 实时 RPM 统计维度调整为 `aggregate_group + model + route_pool + route_group`，默认池与 CLI 专用池中同名真实分组的 RPM 分开展示，避免专用池请求把默认池拓扑节点误点亮。
    - 前端“路由模式配置”中新增“客户端专用流量池”配置区，总开关、Claude CLI 池开关、专用池 targets、专用权重/顺序和回退默认池开关分开展示。

- [x] 聚合分组单渠道子分组组内 retry 修正
  - 背景：
    - 线上 cluster 场景中，一个子分组通常只绑定一个渠道。
    - 原实现把“组内 retry”理解为“当前子分组还有下一档 priority 才继续尝试”，导致单渠道 / 单 priority 子分组失败后直接切到其他子分组。
    - 预期语义应为：`RetryTimes=2` 时，当前子分组首次失败后继续重试当前子分组 2 次，仍失败才切换到下一个子分组。
  - 实现内容：
    - failover 与 cluster 统一使用 `RetryTimes` 作为当前真实分组内部重试预算。
    - 当前真实分组只要仍有可用 channel，即使 retry 次数超过 priority 数量，也继续尝试当前真实分组；底层选渠道会收敛到最后一个可用 priority。
    - 当前真实分组内部预算耗尽后，才进入跨真实分组 fallback。
  - 对应测试：
    - 新增 failover 单渠道 / 单 priority 场景：`RetryTimes=2` 时先尝试当前子分组 3 次，再切下一个子分组。
    - 新增 cluster 单渠道 / 单 priority 场景：`RetryTimes=2` 时先尝试当前子分组 3 次，再按 cluster fallback 选择下一个子分组。
