# 二次开发里程碑事件

## 2026-04-09 | 里程碑：聚合分组链路预算解耦

### 完成事项

- 完成聚合分组链路预算语义调整
  - 聚合分组跨真实分组 fallback 不再受系统全局 `RetryTimes` 提前截断
  - 当前真实分组内部仍按现有 `RetryTimes` 控制 priority/channel 重试
  - 聚合链路可按配置顺序完整尝试 `A -> B -> C`
- 保持现有错误判断规则不变
  - 是否继续下一个真实分组，继续沿用聚合分组 `RetryStatusCodes` 与系统现有重试状态码规则
  - `skip-retry`、本地参数错误、前置校验错误仍直接终止
- 增强聚合 fallback 运行时日志
  - 新增“当前真实分组内部重试”日志
  - 保留“跨真实分组 fallback”与“链路耗尽”日志
- 完成回归验证
  - 补齐聚合分组相关单元测试
  - 本地 `go test ./service ./controller ./model` 通过
  - 本地 `docker build` 与 `docker-compose-dev` 启动验证通过
  - 前端 `bun run build` 通过

### 带来的变化

- 配置了 `A -> B -> C` 的聚合分组后，即使系统 `RetryTimes=0`，仍可完整尝试整条真实分组链路。
- `RetryTimes` 的语义被收敛为“当前真实分组内部预算”，更符合聚合分组高可用链路的预期。
- 普通分组与 `auto` 分组保持原有行为，不会因为本次改造被动改变重试语义。
- 聚合分组的运行日志更容易区分“组内重试”和“跨组 fallback”，便于后续线上排障。

### 备注

- 本次改造不调整 Redis 运行态字段结构，继续使用 `active_index / active_group / last_fail_at / last_success_at / last_switch_at`。
- Redis 聚合运行态 TTL 保持现有实现常量配置，目前为 1 天。

## 2026-04-02 | 里程碑：聚合分组 V1 落地

### 完成事项

- 完成“聚合分组”后端能力建设
  - 新增 `aggregate_groups`、`aggregate_group_targets` 两张表
  - 新增聚合分组管理员 CRUD 接口
  - 新增聚合分组运行时状态存储与懒恢复逻辑
- 完成请求分发链路改造
  - token 仍保持单选 `group`
  - token 可绑定聚合分组
  - 请求进入后按聚合分组展开真实分组链路
  - 支持真实分组优先级顺序、失败切换、恢复后优先尝试高优先级组
- 完成计费逻辑改造
  - 聚合分组使用独立倍率
  - 最终 group ratio 固定取聚合分组倍率
  - 不再跟随底层真实分组倍率变化
  - 异步任务结算复用 group ratio 快照
- 完成用户侧可见性收口
  - 企业用户创建 token 时只看到聚合分组
  - `/api/user/self/groups`、`/api/user/models`、`/api/pricing` 不再向企业用户暴露底层真实分组
  - 用户日志继续记录逻辑组，真实命中组仅保留在管理员调试信息中
- 完成前端管理能力
  - 新增“聚合分组”后台独立菜单
  - 新增聚合分组列表页和创建/编辑面板
  - token 页支持聚合分组选择与聚合标签展示
- 完成测试与文档收口
  - 新增聚合分组相关模型、服务、控制器测试
  - 全量 `go test ./...` 通过
  - 前端 `bun run build` 通过
  - 完成 `aggregate-group-design.md` 与 `迭代开发日志.md`

### 带来的变化

- 平台获得了面向企业客户的“逻辑分组层”，可以在不暴露底层真实渠道组的前提下提供稳定线路。
- 后台可以把多个真实分组聚合成一个对外分组，并通过优先级和 fallback 提高可用性。
- 企业价格体系从“底层真实组倍率驱动”切换为“聚合分组独立倍率驱动”，价格更稳定、可控。
- 存量真实分组、channel、ability、auto 逻辑保持兼容，没有改成 token 多组选路，也没有破坏底层渠道结构。
- 后续如果需要继续迭代模型级 fallback、健康度决策、嵌套聚合策略，已经具备清晰的扩展入口。

### 备注

- 本次里程碑采用自动迁移方案。使用本次二次开发镜像启动时，缺失的聚合分组表会自动创建。
- 后续升级镜像时，迁移逻辑以补齐为主，不会因为正常升级而自动删除现有聚合分组表数据。
