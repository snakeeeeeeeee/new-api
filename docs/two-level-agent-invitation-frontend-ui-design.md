# 邀请功能前端 UI 方案

## 目标

在不改变当前前端视觉风格的前提下，把邀请功能扩展能力补到现有充值页邀请模块中。UI 只服务功能展示和操作，不做新的页面、不做新的设计语言、不引入新的复杂交互。

当前功能入口保持在 `web/src/components/topup/InvitationCard.jsx`。继续使用现有 Semi UI 组件、Tailwind 工具类、卡片布局、圆角 Tag、表格弹窗和 VChart 图表。

## 设计原则

- 保持当前风格：继续使用 `Card`、`Tag`、`Table`、`Select`、`Popconfirm`、`Button`、`Typography.Text`。
- 保持当前层级：邀请信息仍属于充值页中的一个功能卡片，不拆成新菜单或新页面。
- 信息先概览，后明细：先看邀请汇总，再看趋势和被邀请人邀请统计表，最后进入弹窗看完整列表。
- A 和 B 看到不同能力：可开启邀请功能的用户 A 可以开启 B 的邀请功能并查看二级统计；已开启邀请功能的被邀请人 B 只能看自己邀请用户的统计。
- 移动端优先不破坏：列表卡片继续使用 `flex-wrap`，表格使用横向滚动。

## 推荐方案

采用“增强现有 InvitationCard”的方案。

原因：

- 当前邀请相关数据已经集中在充值页，用户认知成本最低。
- 现有组件已经包含邀请码、被邀请人、统计和弹窗列表，新增功能自然落在这里。
- 对路由、导航、权限菜单影响最小，适合先实现功能。

不推荐方案：

- 新增独立邀请管理中心页面：空间更充足，但会增加导航、权限、路由和空状态设计成本，本期不是必要。
- 在管理员邀请码页面混入邀请统计：管理员视角和用户自助视角混在一起，容易造成权限和文案混乱。

## 页面结构

### 1. 邀请统计概览

保留现有顶部统计卡片，继续展示：

- 邀请充值额度
- 邀请实付金额
- 邀请消费额度
- 邀请人数

在概览说明下方增加轻量标签：

- `邀请功能已开启` 或 `邀请功能未开启`
- 可开启邀请功能的用户显示 `可给被邀请人开启邀请码`

这些标签只用于识别状态，不增加新的大块说明。

### 2. 邀请趋势

在概览卡片下方增加一个普通卡片，标题为 `邀请趋势`。

右侧放一个小号 `Select`：

- `按天`
- `按月`

图表使用现有 `@visactor/react-vchart`，展示两条线：

- 充值额度
- 消费额度

数据来源：

- 可开启邀请功能的用户 A：默认展示 A 直属用户 B 的趋势。
- 已开启邀请功能的被邀请人 B：展示 B 直属邀请用户的趋势。
- 未开启邀请功能的用户：可以隐藏趋势卡，或展示空状态 `暂无邀请统计数据`。推荐隐藏，减少无效信息。

### 3. 二级邀请统计

仅可给被邀请人开启邀请码的用户可见。

位置放在 `邀请趋势` 卡片内部下半部分，避免新增太多卡片。

表格字段：

- 用户：`#用户ID 用户名`
- 邀请码
- B 本人充值
- B 本人消费
- B 下级充值
- B 下级消费

表格行为：

- 默认无分页，展示接口返回的被邀请人列表。
- 横向滚动，避免移动端列挤压。
- 提供 `刷新统计` 小按钮，复用现有接口重新拉取数据。

### 4. 我名下的邀请码

保留现有卡片和弹窗。

邀请码预览卡片保持现有信息：

- 邀请码
- 状态
- 目标分组
- 剩余次数
- 邀请人数
- 充值、实付、消费
- 复制邀请链接

给被邀请人自动生成的邀请码不需要特殊样式，只通过后端权限关系控制是否可继续开通。

### 5. 最近被邀请人

保留现有预览卡片和 `全部被邀请人` 弹窗。

增加一项“邀请功能”状态：

- 已开启：绿色 Tag，文案 `邀请功能已开启`
- 未开启且当前用户可给被邀请人开启邀请码：显示 `开启邀请功能` 按钮
- 未开启且当前用户不可给被邀请人开启邀请码：不展示按钮，弹窗表格中可展示普通 `未开启` Tag

开启操作：

- 使用 `Popconfirm` 二次确认。
- 文案保持短句：`确定给该用户开启邀请功能？`
- 说明：`系统会自动生成一个零奖励的邀请码。`
- 成功后刷新当前用户信息和邀请统计。

### 6. 说明区

保留当前说明卡片，补充但不展开长文：

- 新统计只统计新邀请码体系。
- 邀请码删除或禁用后，历史归属和统计仍保留。
- 如需管理员分配新邀请码，联系管理员。

## 状态设计

### 加载中

- 统计接口加载时，图表区域和二级表格使用 Semi 的 `loading` 或保留空图表。
- 开启邀请功能时，只让当前行按钮 loading，不阻塞整个页面。

### 空数据

- 无邀请码：沿用 `您当前还没有绑定的邀请码。`
- 无被邀请人：沿用 `暂无被邀请人记录。`
- 可给被邀请人开启邀请码的用户还没有给任何被邀请人开启邀请码：被邀请人邀请统计表展示空表即可，不额外强调。
- 无代理能力：隐藏邀请趋势和二级邀请统计，只保留基础邀请统计。

### 错误

- 接口错误统一使用现有 `showError(message)`。
- 开启失败展示后端返回文案，例如无权限、非直属用户、目标用户不允许开通。

## 数据流

页面加载时：

1. 父组件继续获取 `/api/user/self`。
2. `InvitationCard` 根据 `invite_agent_level` 和 `can_grant_invitation` 判断是否展示邀请功能模块。
3. 需要展示邀请功能模块时，请求 `/api/user/self/invite_agent_stats?period=day`。

切换日/月：

1. 更新本地 `agentStatsPeriod`。
2. 重新请求 `/api/user/self/invite_agent_stats?period=day|month`。
3. 更新图表和被邀请人邀请统计表。

开启邀请功能：

1. 点击被邀请人行的 `开启邀请功能`。
2. 确认后调用 `POST /api/user/self/invitees/:id/enable_invitation`。
3. 成功后刷新 `/api/user/self` 和统计接口。

## 响应式

- 顶部四个指标保持 `grid-cols-2 md:grid-cols-4`。
- 标签和按钮使用 `flex-wrap`。
- 被邀请人邀请统计表和被邀请人弹窗表格使用 `scroll={{ x: 'max-content' }}`。
- 移动端预览卡片中按钮放在标签流末尾，不另起复杂操作栏。

## i18n

所有新增文案继续使用中文 key：

- `邀请功能已开启`
- `可给被邀请人开启邀请码`
- `邀请趋势`
- `按天`
- `按月`
- `二级邀请统计`
- `被邀请人充值`
- `被邀请人消费`
- `其邀请用户充值`
- `其邀请用户消费`
- `邀请功能已开启`
- `开启邀请功能`
- `确定给该用户开启邀请功能？`
- `系统会自动生成一个零奖励的邀请码。`

新增或调整文案后运行：

```bash
cd web
bun run i18n:extract
bun run i18n:sync
```

`i18n:lint` 如仍失败，需要区分本次新增问题和历史问题。

## 测试方案

### 手动测试

- 普通用户：不显示开启邀请码按钮，不显示二级邀请统计。
- 可开启邀请功能的用户 A：显示邀请功能状态和开启按钮，可以给直属 B 开启邀请功能。
- 已开启邀请功能的被邀请人 B：显示自己的趋势，不显示开启按钮，不显示二级邀请统计。
- A 给 B 开启后，列表状态变为已开启，统计刷新后出现 B 行。
- B 邀请 C 后，B 的趋势和汇总包含 C，A 的被邀请人邀请统计中 `其邀请用户` 包含 C。

### 自动测试

本期前端可先不新增复杂 UI 单测。后端 model/controller 已覆盖核心数据统计和权限限制。前端构建至少运行：

```bash
cd web
NODE_OPTIONS=--max-old-space-size=4096 bun run build
```

## 实施顺序

1. 调整 `InvitationCard.jsx` 的邀请功能模块展示条件，避免未开启邀请功能用户看到空趋势。
2. 补齐开启邀请按钮在预览卡片和弹窗表格中的状态。
3. 补齐趋势图和被邀请人邀请统计表的 loading、空状态。
4. 运行 i18n extract/sync。
5. 运行前端构建和 Docker API 回归。
